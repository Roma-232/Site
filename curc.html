Основы тестирования: QA, QC и жизненные циклы
2 часа

Теория урока

QA (Quality Assurance) — обеспечение качества, процесс-ориентированный подход для предотвращения дефектов.

QC (Quality Control) — контроль качества, продукт-ориентированный подход для выявления дефектов.

Жизненные циклы:

  • SDLC — Software Development Life Cycle: анализ, проектирование, разработка, тестирование, внедрение
  • STLC — Software Testing Life Cycle: планирование, проектирование тестов, выполнение, завершение

Практический тест

Проверьте свои знания по пройденной теме:

1. Что такое QA (Quality Assurance)?

2. Чем отличается SDLC от STLC?

3. Кто обычно занимается QC (Quality Control)?

Виды и уровни тестирования ПО
2.5 часа

Теория урока

По доступу к коду:

  • Чёрный ящик — тестирование без знания внутренней структуры
  • Белый ящик — тестирование с доступом к коду
  • Серый ящик — частичное знание внутренней структуры

Уровни тестирования:

  1. Модульное (Unit)
  2. Интеграционное (Integration)
  3. Системное (System)
  4. Приемочное (Acceptance)

Практический тест

Определите вид тестирования для каждого сценария:

1. Тестировщик проверяет работу формы входа, не зная как реализована проверка пароля.

2. Разработчик пишет тесты для отдельной функции calculatePrice().

3. Заказчик проверяет готовый продукт перед запуском.

Тест-дизайн: Эквивалентное Разделение и BVA
3 часа

Теория урока

Тест-дизайн — создание тестовых случаев с использованием специальных техник.

Основные техники:

  • Эквивалентное Разделение (EP) — группировка входных данных в классы
  • Анализ Граничных Значений (BVA) — тестирование значений на границах диапазонов

Пример: Поле "Возраст" от 18 до 65 лет.

Валидные: 18, 19, 64, 65

Невалидные: 17, 66

Практический тест

Примените техники тест-дизайна:

1. Поле принимает значения от 1 до 100. Какие значения выбрать для BVA?

2. Для поля "Процент скидки" (5-20%) сколько классов эквивалентности?

3. Какая техника помогает сократить количество тестов?

Чек-листы и тест-кейсы: документация
2 часа

Теория урока

Чек-лист — список пунктов для проверки (что проверить).

Тест-кейс — детальная инструкция с шагами (как проверить).

Структура тест-кейса:

  1. ID и название
  2. Предварительные условия
  3. Шаги выполнения
  4. Ожидаемый результат
  5. Фактический результат

Практический тест

1. Когда использовать чек-лист вместо тест-кейса?

2. Что должно быть в ожидаемом результате тест-кейса?

Баг-репорты: описание дефектов
2.5 часа

Теория урока

Баг-репорт — документ, описывающий найденный дефект.

Обязательные поля:

  • Заголовок (кратко и понятно)
  • Шаги воспроизведения
  • Фактический результат
  • Ожидаемый результат
  • Окружение (браузер, ОС, версия)
  • Серьезность (Severity)

Практический тест

1. Какая серьезность (Severity) у бага: "Приложение крашится при нажатии кнопки Сохранить"?

2. Что важнее для шагов воспроизведения?
